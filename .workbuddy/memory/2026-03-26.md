## EntityManager 遮挡渲染改为两遍绘制（07:55）

### 问题
player(3,6) 被 megatower 遮挡时，megatower 触发半透明正确，但绘制顺序错误：
megatower（半透明）被画在 player 之前，player 不透明地覆盖在建筑上面，遮挡效果失效。

### 根本原因
单遍 depth 排序无法同时满足所有遮挡方向：
- player 在建筑 SE 方向：player depth > buildingNW → 建筑先画 player 后画 → 不需要建筑覆盖（走另一路径）
- player 在建筑 W/N/NW 方向：player depth > buildingNW → 同样建筑先画 player 后画 → **但这里需要建筑覆盖 player！**

### 修复（EntityManager.ts，两处）
`renderWithOcclusionSystem` 和 `renderWithInternalOcclusion` 均改为**两遍绘制**：
- **Pass 1**：按 NW 锚点 depth 排序，画所有非半透明建筑 + 所有 character（不透明）
- **Pass 2**：画所有半透明遮挡建筑（按 SE 角 depth 互相排序），最后绘制，天然覆盖在 player 上面

### 结果
- player(3,6) 先画（Pass1），megatower 半透明后画（Pass2）覆盖 → 透过建筑看到 player ✅
- player(4,7) 不在 occlusionMap → megatower 不进入 Pass2 → 正常排序 player 在前 ✅

- player(3,6) depth=9，megatower SE depth=12 → 12>9 ✓

- player(3,3)：(4,4)→NW→(3,3) ✓ → 遮挡 ✅

## EntityManager 深度排序 Bug 修复（07:50）

### 问题
player(4,7) 被 megatower(col:4,row:4, 3×3格, height:140) 遮盖一角。

### 根因
`renderWithOcclusionSystem`（和 `renderWithInternalOcclusion`）的排序一律用"SE角 depth"：
- megatower SE角 = (6,6)，depth = 12
- player(4,7) depth = 11
→ megatower 排在 player 之后绘制，直接覆盖 player 右上角

而 `semiTransparentBuildings` 集合只在 `isOccluded` 返回 true 时才加入建筑，
player(4,7) 不在 megatower 的 occlusionMap 里，所以 megatower 以 alpha=1.0 画在 player 上面。

### 修复（EntityManager.ts）
修改排序策略为**分情况比较**：
- 建筑 vs 建筑：用 SE 角（保证大建筑互相排序正确）
- 角色 vs 角色：用 NW 锚点 col+row
- 建筑 vs 角色（混合）：**统一用 NW 锚点** ← 关键修复
  - megatower NW = 4+4=8，player NW = 4+7=11 → megatower 先画，player 后画 ✅

同时修复了 `renderWithInternalOcclusion`（legacy 路径）的相同问题。

### 验证
- player(4,7)：megatower NW depth=8 < player depth=11 → megatower 先画，player 覆盖其上 ✅
- player(5,6) 在建筑 SE 方向：仍然正确被遮挡 ✅

## 鼠标高亮提前跳格 Bug 修复（07:02）

### 根本原因

`InputManager.screenToWorld` 存在**两个独立的严重错误**：

1. **逆变换公式错误**：自写的等轴测逆变换忘记除以 `tileScale`（即 `Projection.tileScale`，由 cellSize 决定），导致坐标比例始终不对
2. **硬编码角度**：写死了 30°，不使用 `Projection` 里配置的 `viewAngle`，一旦角度改变立刻错位

同时 `GridSystem.worldToGrid` 和 `InputManager.worldToGrid` 都用 `Math.round` 而非 `Math.floor`：
- `gridToWorld(col, row)` 返回的是格子**左上角**坐标 `(col * tileW, row * tileH)`
- 鼠标在格子前半段时 round 会映射到上一个格子，造成高亮"提前跳"

### 修复内容

**`src/input/InputManager.ts`**
- `screenToWorld`：废弃自写逆变换，改为调用 `camera.screenToWorld(sx, sy, projection, parallaxFactor)` —— 使用 IsoCamera 中经过 Projection.screenToWorld 的正确逆矩阵
- `worldToGrid`：`Math.round` → `Math.floor`
- `onClick`：直接调用 `camera.screenToWorld`，不再走 `this.screenToWorld`

**`src/world/GridSystem.ts`**
- `worldToGrid`：`Math.round` → `Math.floor`，加详细注释说明原因

## OcclusionSystem 遮挡方向全面修正（07:46）

### 坐标系
- 原点 (0,0) 在西北，col 向东北增大（屏幕右下），row 向西南增大（屏幕左下）
- depth = col+row，越大越靠前景（越靠屏幕下方，后绘制压在前面）

### 遮挡逻辑正确定义
等轴测视角下，建筑遮挡 character 的充要条件：
1. 建筑 SE 角 depth > character depth（建筑在前景）
2. character 在建筑的**正东南方向**（col 和 row 都更大）才真正被压在背后

E(+1,0) 和 S(0,+1) 是建筑两侧可见面，不产生遮挡。只有 SE(+1,+1) 方向投影才正确。

### 两处修改
1. `calculateBuildingOcclusion` directions 从 `[NW, W, N]` → 改为只有 `[SE(+1,+1)]`
2. `getOccludingBuildings` 深度过滤：`occ.depth > entityDepth`（建筑比 entity 更靠前景才遮挡）

### 验证
- player(4,7)：megatower 阴影格 = {(5,5)(6,6)(5,6)(6,7)(5,7)}，(4,7) 不在 → VISIBLE ✅
- player 走到 (5,6) 等正东南格时：在阴影内 + depth 判断通过 → OCCLUDED ✅

## OcclusionSystem.getOccludingBuildings 遮挡方向 bug 修复（07:42）

### 坐标系确认
- 原点 (0,0) 在**西北角**，col 向东北增大，row 向西南增大
- depth = col + row：值越大越靠东南（前景），值越小越靠西北（背景）
- 摄像机从东南往西北看，建筑遮挡其**西北侧**（depth 更小）的实体

### Bug 根因
`getOccludingBuildings` 直接返回 occlusionMap 里的原始数据，没有过滤"建筑在 entity 前景"的错误情形。建筑脚印内（及其西北邻格）的阴影格与建筑自身格重叠，导致 player 站在建筑东南角（如 megatower 的 (6,6) 格）时被误判为遮挡。

### 修复
`OcclusionSystem.getOccludingBuildings` 增加深度过滤：
```ts
return raw.filter(occ => occ.depth < entityDepth);
```
只保留 SE 角 depth **严格小于** entity depth 的建筑，确保只有真正在背景中的建筑才算遮挡。

## 运行时崩溃修复：entity.isBuilding is not a function（07:20）

### 问题根因

demo 用纯 JS 对象字面量 `{ id, col, row, height, ... }` 调用 `entityManager.addEntity()`。TypeScript duck typing 让这在编译期不报错，但运行时对象没有 `Entity` 原型上的 `isBuilding()` / `isCharacter()` 方法，导致 `OcclusionSystem.calculateOcclusionMap` 崩溃。

- 检测 `isBuilding` 是否为函数，若不是则动态注入方法
- 当 `entityType` 未设置时，用 `height >= 50` 启发式自动推断（向后兼容旧 demo）
- 对正规 `Entity` 子类实例**零开销**（if 直接跳过）
